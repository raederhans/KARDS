# KARDS Product Direction Research Plan

## Goal

基于当前仓库事实、公开用户问题、成熟产品和活跃开源项目，确定 KARDS Card Forge 下一阶段最值得发展的方向，以及可以安全借鉴的模块与项目。

## Scope

- 当前产品边界、主要用户旅程、已交付能力和现有路线图。
- 公开可验证的同类产品体验、常见用户摩擦和可复用交互模式。
- 与当前 React、Canvas、纯本地静态应用架构相容的开源模块或实现思路。
- 未来一周、未来一季度和长期探索三个时间层级的优先级建议。

## Sources of truth

- 当前 `main`、`origin/main` 指针、最近十个提交和 worktree registry。
- `docs/active/roadmap.md`、`lessons learned.md`、归档任务证据与当前源码、测试。
- 官方产品文档、官方仓库、维护者文档和可验证的公共讨论。

## Stages

- [x] Stage 1: 建立本地产品、用户流程、架构和交付状态基线。
- [x] Stage 2: 研究外部用户问题、成熟产品模式和候选模块。
- [x] Stage 3: 比较方案并形成按价值、成本、风险排序的路线图。
- [x] Stage 4: 独立复核证据、完成研究简报和留档。

## Acceptance criteria

- 每个高优先级建议都有本地证据和明确用户目标。
- 外部事实使用当前可访问的一手来源，并区分事实、公共轶事与推断。
- 至少给出可执行的本周、一季度和长期方向。
- 候选模块或项目说明可借鉴部分、集成边界、主要风险和不应照搬之处。
- 不修改生产代码，不扩大现有资源权利、账号、网络内容或 gameplay 边界。

## Non-goals

- 本轮不实现功能、不新增依赖、不发布版本。
- 不把单条抱怨当作高频需求，不承诺未经验证的市场规模。
- 不重新设计整个 Canvas renderer，也不复制受限制的 KARDS 资源。

## Risks and constraints

- 项目自身公共讨论可能很少，必须降低“用户频率”结论的置信度。
- 同类工具的功能边界、许可和资源权利不同，只能借鉴交互或架构模式。
- 当前应用明确是本地单卡工具；推荐不能默认引入账号、云端或牌组平台。
